1.  PROJECT OVERVIEW
====================
A Hebrew city-ranking site analyzing ideological influence in Israeli
education enrichment programs (Gefen marketplace).  The site ranks ~20 cities
by an "evidence score" combining three pillars:
  - Education programs (Gefen data, reviewed by humans + AI)
  - Municipal protocol keyword analysis (Pole A vs Pole B orientation)
  - News signal analysis (Google News RSS, progressive-topic counts)

The main entry point is index.html — a single-page app with 6 tabs, all
rendered client-side from JS data files loaded via <script> tags.

7.  DATA GAPS & UNLINKED DATA
==============================

7.1  KNOWN GAPS
---------------
  - org_key is empty ("") for most programs in DATA.rankings[].proven_programs[].
    Only programs whose org was investigated in deep research have an org_key.
    This means buildProgramCard() must fall back to fuzzy name matching via
    findOrgProfile() and findOrgReport().

  - Not all programs in worst_programs{} have a matching entry in program_info{}.
    When no match exists, the program card lacks enriched detail (gemini
    description, ideology markers, etc.).

  - SUSPICIOUS_CONCEPTS.program_concepts{} only covers ~30 programs with explicit
    concept mappings.  The rest depend on keyword_rules[] regex fallback, which
    only works if the program has a research_desc or gefen_description.

  - RESEARCH.program_cluster_map{} only covers ~16 programs.  The rest rely on
    program_name_patterns{} (fuzzy name matching), which can miss programs.

  - protocol_findings_ids[] in rankings entries reference PROTOCOL_FINDINGS IDs,
    but only 5 cities have findings (23 total findings out of 537 protocols
    scanned).  10 cities have zero findings.

  - data/news/{CITY_KEY}.json files exist for ~20 cities but are only loaded
    on-demand when the user clicks "load news" in a city detail panel.

7.2  UNUSED / STANDALONE DATA FILES
------------------------------------
  These files exist in data/ but are NOT loaded by index.html at runtime:

  - data/gefen_metadata.json       -- source only, consumed by enrich script
  - data/manual_metadata.json      -- DB export, not directly rendered
  - data/program_lookup.json       -- reference lookup, not rendered
  - data/programs_subset.json      -- reference subset, not rendered
  - data/merged_city_ranking.json  -- pre-computed rankings (data.js has final version)
  - data/city_details.json         -- protocol details (data.js has protocol_details{})
  - data/methodology_computations.json  -- stats (rendered by data/methodology.html)
  - data/acri_research.json        -- standalone ACRI dossier (not rendered in main site)
  - data/research_efshar_acheret.json -- standalone dossier (not rendered in main site)
  - data/research_lens.json        -- framework definition (rendered by data/research_lens.html)
  - data/city_news_articles.json   -- full news corpus (news tab loads per-city JSONs instead)
  - data/city_news_compact.json    -- compact news counts (index.json is same structure)

  Effectively, the data/ folder contains both source materials for the build
  pipeline AND standalone viewer pages.  The main site (index.html) uses only
  the 6 .js files at the project root.

7.3  POTENTIAL LINKAGE IMPROVEMENTS
------------------------------------
  a) Many programs in worst_programs{} lack .id values that exist in
     program_info{}.  A normalization pass could match by name to add IDs.

  b) org_key could be auto-populated for any org name that matches a
     RESEARCH_ORGS.orgs entry (currently only ~10% have org_key set).

  c) The 7 RESEARCH clusters are linked to programs via only 16 explicit
     mappings + ~30 name patterns.  Many programs operated by cluster orgs
     are not linked (e.g., a program by "דרור ישראל" might not match if
     the pattern list misses its specific name).

  d) SUSPICIOUS_CONCEPTS explicit mapping covers ~30 programs; there are
     likely 50+ more programs that should have explicit concept tags based
     on their research_desc content.

  e) The news data (data/news/*.json) is loaded lazily per city. The
     news_pillar_score and news_actions_detail in DATA.rankings are
     pre-computed, but only 3 strong signals per city are stored in
     rankings. The full article corpus is much larger.
